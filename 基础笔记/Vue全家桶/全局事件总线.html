# 5.1全局事件总线1

  • 全局事件总线：通过一个傀儡实现任意组件间通信

1、X作为傀儡，A先给X绑定一自定义事件demo，回调函数自然写到了A里；

2、之后D想对A传递信息，需要触发（$emit）demo事件，然后传递参数“666”

3、既然回调函数在A里，且触发事件后就执行回调函数，那A自然而然就会收到参数

  • 那这个X需要什么条件呢、

1、得保证X得让所有组件都看见它

2、既然X能绑定事件，那它就得能调用$on、$off、$emit

  • 具体实现

1：想法

​ （1）使用window，给window绑定属性,因为不管用什么框架，都脱离不了window，但这样写不太好

​ window.x = {a:1,b:2}

​ （2）VueComponent，想往VueComponent这个原型对象上放上一x，那所有的组件不都能看见了吗？但是又不存在VueComponent.prototype.x = {}，因为它不是一个直接存在的东西,复习一下：

关于VueComponent：

1.school组件本质是一个名为VueComponent的构造函数，且不是程序员定义的，是Vue.extend生成的。

2.我们只需要写或，Vue解析时会帮我们创建school组件的实例对象，

即Vue帮我们执行的：new VueComponent(options)。

3.特别注意：每次调用Vue.extend，返回的都是一个全新的VueComponent！！！！

4.关于this指向：

​ (1).组件配置中：

​ data函数、methods中的函数、watch中的函数、computed中的函数 它们的this均是【VueComponent实例对象】。

​ (2).new Vue(options)配置中：

​ data函数、methods中的函数、watch中的函数、computed中的函数 它们的this均是【Vue实例对象】。

5.VueComponent的实例对象，以后简称vc（也可称之为：组件实例对象）。

​ Vue的实例对象，以后简称vm。

（3）1.一个重要的内置关系：VueComponent.prototype.proto === Vue.prototype

​ 2.为什么要有这个关系：让组件实例对象（vc）可以访问到 Vue原型上的属性、方法。

往Vm身上放东西，vc能看见

vue只有一个，vue的原型对象vm也只有一个

所以在引入vue的地方 main.js里写vm.prototype.x = {}

2、想法

  • 要知道$on,$off,$emit都在Vue的原型对象里 （在vue.prototype里），且vm和vc都能够使用

  • 为了能使用$on,$off,$emit 可以Vue.prototype.x = vc当然不能直接写

# 5.2全局事件总结

  1. 一种组件间通信的方式，适用于任意组件间通信。

  2. 安装全局事件总线：

    new Vue({
   ......
   beforeCreate() {
      Vue.prototype.$bus = this //安装全局事件总线，$bus就是当前应用的vm
   },
    ......
})

  3. 使用事件总线：

    1. 接收数据：A组件想接收数据，则在A组件中给$bus绑定自定义事件，事件的回调留在A组件自身。

      methods(){
  demo(data){......}
}
......
mounted() {
  this.$bus.$on('xxxx',this.demo)
}

    2. 提供数据：this.$bus.$emit('xxxx',数据)

  4. 最好在beforeDestroy钩子中，用$off去解绑当前组件所用到的事件。

# 前端框架的引导理解

使用前端框架的逻辑——>程序员把数据维护好，那至于说页面怎么变，就是vue的功劳了，数据驱动着页面的显示，程序猿只需要维护好数据，页面怎么变那是vue的事；就像input框里的勾，勾不勾选对于我们来说就是todolist的todo.done值

<script>
	import pubsub from 'pubsub-js'
	export default {
		name:'MyItem',
		//声明接收todo
		props:['todo'],
		methods: {
			//勾选or取消勾选
			handleCheck(id){
				//通知App组件将对应的todo对象的done值取反
				// this.checkTodo(id)
				this.$bus.$emit('checkTodo',id)
			},
			//删除
			handleDelete(id){
				if(confirm('确定删除吗？')){
					//通知App组件将对应的todo对象删除
					// this.deleteTodo(id)
					// this.$bus.$emit('deleteTodo',id)
					pubsub.publish('deleteTodo',id)
				}
			},
			//编辑
			handleEdit(todo){
                
                // 1、如果在这个函数里只写
                // todo.isEdit = true，看着像是给todo加上一个isEdit属性，再让两个v—show相互取反，但实际这个属性加上去后没有对应的getter和setter，这个数据不是响应式的，这个数据的添加和修改不会引起vue的重新渲染
                // 2、这么写
                // this.$set(todo,'isEdit',true)响应式添加，可以被vue认可
                
				if(todo.hasOwnProperty('isEdit')){
                    // 借助一个判断todo属性的一个api hasOwnProperty
					todo.isEdit = true
				}else{
					// console.log('@')
					this.$set(todo,'isEdit',true)
				}
			},
			//失去焦点回调（真正执行修改逻辑）意思就是回车或者鼠标点击其他地方后，编辑完成
			handleBlur(todo,e){
				todo.isEdit = false
				if(!e.target.value.trim()) return alert('输入不能为空！')
                // 直接用全局事件总线写
				this.$bus.$emit('updateTodo',todo.id,e.target.value)
                // 把要修改的todoid递过去，把要修改的最新的value值递过去
                // 这里不能写todo.title，这个值还是原来的值，不是输入框里的值；给input框绑定的handleblur，可以通过todo,$event这个参数拿到此时的这个事件对象
			}
		},
	}
</script>

# 5.5$nextTick

  • 下一轮，其可以指定一个回调，且这个回调会在dom节点更新完之后执行
  • 其作用替代了setTimeOut但是比setTomeout更加的官方
handleEdit(todo){
				if(todo.hasOwnProperty('isEdit')){
					todo.isEdit = true
				}else{
					// console.log('@')
					this.$set(todo,'isEdit',true)
				}
    			// isEdit改变后不会立即重新渲染，因此把this.$refs.inputTitle.focus()直接放在这里，因为input框还是隐藏的状态，所以这里也获取不到，因此使用$nextTick方法
				this.$nextTick(function(){
					this.$refs.inputTitle.focus()
				})
			},

  1. 语法：this.$nextTick(回调函数)

  2. 作用：在下一次 DOM 更新结束后执行其指定的回调。

  3. 什么时候用：当改变数据后，要基于更新后的新DOM进行某些操作时，要在nextTick所指定的回调函数中执行。
